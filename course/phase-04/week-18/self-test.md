# Week 18 自测：ILP 极限

## Q1：收益递减

某超标量处理器发射宽度从 2 增加到 4 时 IPC 从 1.5 提升到 2.1。从 4 增加到 8 时 IPC 仅从 2.1 提升到 2.4。计算各自的提升幅度（百分比），解释为什么第二个提升远小于第一个。

## Q2：复杂度墙

假设发射宽度为 w 的 wakeup+select 逻辑复杂度为 \(O(w^2)\)。宽度从 4→8 的复杂度增加倍数是 4→2 的多少倍？

## Q3：Phase 4 综合匹配

将左侧技术与其核心贡献连线：
- A) Scoreboard     →  ① 推测执行 + 精确异常
- B) Tomasulo       →  ② 每周期多条指令
- C) ROB            →  ③ 乱序执行（但 WAR 阻塞）
- D) Superscalar    →  ④ 寄存器重命名消除 WAW/WAR
- E) VLIW           →  ⑤ 编译器静态调度 ILP

## Q4：ILP 墙

某个 benchmark 的理论 ILP 上限为 8。但即使无限 ROB、无限发射宽度、完美 Cache、完美分支预测，实际 IPC 也只有 5.2。列举 3 个可能的原因。

## Q5：全栈分析

给定以下代码片段：
```c
for (i = 0; i < 1000; i++) {
    a[i] = b[i] * c[i] + d[i];  // vector multiply-add
}
```
分析从哪些维度提升 ILP：编译器优化、动态调度、多发射、SMT/VLIW，预估理想情况下的 IPC。

---

## 答案

### A1
- 2→4：提升 (2.1-1.5)/1.5 = 40%
- 4→8：提升 (2.4-2.1)/2.1 = 14.3%
- 原因：程序 ILP 有限。发射宽度增大后，更多周期发射槽无法被填满（指令窗口内的可用 ILP 早已被挖掘完毕）。这是 ILP 墙的体现。

### A2
- \(4^2 : 2^2 = 16:4 = 4x\)
- \(8^2 : 4^2 = 64:16 = 4x\)
- 绝对增量：8 宽度比 4 宽度的硬件开销增加了 64-16=48 单位，而 4 比 2 只增加 12 单位。复杂度的绝对增量是 4 倍。

### A3
A→③, B→④, C→①, D→②, E→⑤

### A4
1. 依赖链长度：关键路径上的串行依赖（如循环间依赖）限制了 ILP
2. 指令混合：分支、load/store 等无法完全并行化
3. 存储别名分析不完美：编译器/硬件无法证明两个地址不冲突，保守地串行化

### A5
- 编译器：循环展开 + 软件流水，减少循环开销
- 动态调度：乱序执行 + ROB 大窗口，多迭代并行
- 多发射：每条指令（load b, load c, mul, load d, add, store a）6 种操作，至少需要 6 发射
- SMT：不适用（单线程），VLIW：完美场景，编译器可静态调度
- 理想 IPC：在有 2 load、1 store、1 mul、1 add FU 且足够发射宽度下，每次迭代 6 条指令，理想 CPI ≈ 1-2（受限于 load/store 带宽）
