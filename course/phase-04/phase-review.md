## 答案

### 选择题
1. **D**（Write Result 阶段检查 WAR：目标寄存器是否还有指令需要读）
2. **C**（寄存器重命名将 WAW 转换为对不同物理寄存器的写入）
3. **C**（寄存器重命名由 Tomasulo 的保留站 + reg_status 实现，ROB 负责按序提交）
4. **B**（1.2/4=30%）
5. **C**（SMT 同一周期发射不同线程的指令，充分利用空闲发射槽）
6. **B**（VLIW bundle 所有 slot 同步执行，任一 slot stall 则整个 bundle stall）
7. **B**
8. **B**（程序自身 ILP 已是瓶颈，增大 ROB 帮助有限，受 ILP 墙限制）

### 分析题

**9.**
- C1: MULT→Mul0, reg[F0]=Mul0; ADD→Add0, reg[F2]=Add0
- C2: ADD 执行完(C3)，Write→CDB
- C3: 第三条 ADD→Add1, Qj=Mul0, Qk=Add0
- C4: Add0 Write → CDB，Add1 捕获 F2 的值（Vk 就绪）
- C4-12: 等待 Mul0
- C13: Add1 Execute, C14: Write
- 总 14 周期

记分板：更慢。ADD F2 的 Issue 会因 WAW（MULT 占用 F2? 不，MULT 写 F0 不是 F2。但 ADD F10 的源是 MULT 的 F0——RAW，记分板需要 MULT 完成后 ADD F10 才能读数。Tomasulo 中 ADD F10 可以提前等待 CDB 广播。）记分板处理 WAR（如果有的话）需要 stall Write。

**10.**

| 场景 | Scoreboard | Tomasulo | Tomasulo+ROB |
|------|-----------|----------|--------------|
| RAW (ALU→ALU) | ReadOps 等待 | 标签等待 CDB | 标签等待 CDB |
| WAW | Issue 阻塞 | 重命名消除 | 重命名消除 |
| WAR | Write 阻塞 | 重命名消除 | 重命名消除 |
| 推测执行 | 不支持 | 不支持 | 支持 |
| 精确异常 | 不支持 | 不支持 | 支持（Commit 阶段检测） |
| 误预测恢复 | 不支持推测 | 不支持推测 | 清空 ROB，恢复状态 |

**11.**
- 额外 CPI = 0.2 × 0.08 × 4 = 0.064（但双发射下，误预测时 flush 4×2=8 条指令，比单发射损失更多。更准确：flush 损失的指令数 = 4×2=8 条，实际额外 CPI ≈ 0.2×0.08×4×2 = 0.128）
- 4 发射下：flush 损失 4×4=16 条，额外 CPI ≈ 0.2×0.08×4×4=0.256
- 发射宽度越大，同样的误预测惩罚（cycles）导致更多指令被 flush → 额外 CPI 更大

**12.**
- **依赖链**：整数程序存在大量短依赖链（如循环计数器的 increment→compare→branch），形成串行瓶颈。Wall 的研究中即使消除控制依赖（完美预测）和存储依赖（完美 alias），数据依赖链仍限制了 ILP。
- **存储别名**：整数程序有大量 load/store，完美 alias 分析假设所有 load/store 不冲突——实际做不到。硬件保守处理（store 后的 load 可能依赖），限制了并行度。
- **分支**：即使是完美预测，整数程序的短基本块（平均 4-6 条指令）意味着每个基本块的 ILP 受块内依赖限制。跨基本块的并行需要推测，但完美预测消除了此限制后，数据依赖仍是瓶颈。
