# Week 2 自测

## Q1：平均方式的选择

两个基准程序在两台机器上的性能比（B/A）为：0.1 和 10.0。
- 算术平均 = ?
- 几何平均 = ?
- 哪个更好地反映了"典型"加速比？为什么？

## Q2：SPEC 分数解释

处理器 X 的 SPECspeed2017_int_base = 8.5。这意味着什么？它的性能是参考机器的多少倍？

## Q3：MIPS 批判

处理器 A：1 GHz，CPI=1.0，每条指令做 1 个有用操作。
处理器 B：1 GHz，CPI=2.0，但每条指令做 5 个有用操作（VLIW风格）。
哪个 MIPS 更高？哪个实际有用操作的吞吐率更高？这说明什么？

## Q4：基准测试陷阱

编译器开发者发现：在 SPEC 基准测试中，某个 benchmark 80% 的时间花在一个可以被完全向量化的循环上。编译器团队专门为这个循环写了一个优化 pass，让这个 benchmark 加速 5x。整体 SPEC 分数因此提升了 15%。市场部大肆宣传"性能提升 15%"。这个宣传有什么问题？

## Q5：能效计算

处理器 A：性能 100，功耗 50W。
处理器 B：性能 80，功耗 25W。
哪个能效更高？如果你运营一个数据中心，10 万颗处理器，你会选哪个？为什么？

---

## 答案

### A1
- 算术平均 = (0.1+10.0)/2 = 5.05
- 几何平均 = sqrt(0.1×10.0) = 1.0
- 几何平均更好，因为性能比是乘法性质的。A/B 的算术平均 ≠ 1/(B/A 的算术平均)，但几何平均有这个对称性。

### A2
SPECspeed2017_int_base = 8.5 意味着该处理器执行整数基准测试的速度是参考机器的 8.5 倍。"base"表示使用保守的编译器优化。

### A3
- 处理器 A MIPS = 1000/1.0 = 1000
- 处理器 B MIPS = 1000/2.0 = 500
- A 的 MIPS 更高。但 B 每秒完成的操作 = 1000MHz × 5/2.0 = 2500 Mops，而 A 只有 1000 Mops。B 实际更快，但 MIPS 更低。MIPS 忽略了每条指令的"工作密度"。

### A4
这是"benchmark specialisation"陷阱。编译器团队针对特定 benchmark 做了特殊优化，提升的 15% 可能只适用于这一个 benchmark，对真实应用无效。更糟糕的是，如果这个优化在非 SPEC 代码上反而有负面影响，真实性能可能下降。

### A5
- A 能效 = 100/50 = 2.0 perf/W
- B 能效 = 80/25 = 3.2 perf/W → B 能效更高
- 数据中心选 B：10万颗 × 25W = 2.5MW vs 5MW，电费差巨大。即使单颗性能低 20%，同等功耗可部署 2 倍数量，总吞吐量 1.6x。
