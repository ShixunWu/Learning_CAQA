## 设计审查问题

### 通用问题

**Q1.** 你的项目解决了什么具体的量化问题？请用一句话表述。

**Q2.** 你计划如何建立基线（Baseline）？基线数据将是你的比较基准。

**Q3.** 你的实验设计中的自变量（Independent Variables）和因变量（Dependent Variables）分别是什么？

**Q4.** 你如何确保实验结果的统计显著性？样本量是多少？

### 项目特定问题

**项目 1 (gem5 CPU)**:
- Q5. 你计划改变哪些微架构参数？参数范围有上限吗（如 ROB 大小 32→256 vs 32→1024）？
- Q6. gem5 模拟速度可能很慢（~100K instructions/sec）。你计划如何管理模拟时间？

**项目 2 (缓存优化)**:
- Q5. 你如何分离 Compulsory/Capacity/Conflict Miss？用什么工具或方法？
- Q6. 预取器设计如何平衡覆盖率和准确性？

**项目 3 (GPU Kernel)**:
- Q5. 你的 roofline 模型基于什么硬件参数？（计算峰值、内存带宽）
- Q6. 你计划使用哪些 CUDA Profiler 指标来诊断瓶颈？

**项目 4 (NoC)**:
- Q5. 你测试哪些流量模式？（Uniform Random, Transpose, Hotspot, Bit Complement）
- Q6. 如何定义和测量 NoC 的饱和吞吐量？

**项目 5 (DNN 加速器)**:
- Q5. 你分析哪些 DNN 模型？为什么选择它们？
- Q6. 你如何建模不同存储层级（DRAM/SRAM/RF）的能耗？

## 答案

### 设计审查参考回答

**Q1.** 示例："通过在 gem5 中扫描 ROB 大小和发射宽度，量化超标量参数对 SPEC CPU 2017
中 5 个 benchmarks 的 IPC 影响，找出 Pareto 最优配置。"

**Q2.** 基线是最简配置的实现：如项目 1 用顺序单发射 MinorCPU；项目 2 用 32KB 8-way L1
无 L2；项目 3 用 naive 三重循环 CUDA kernel。

**Q3.** 
- 自变量：微架构参数（发射宽度、缓存大小、数据流策略等）
- 因变量：性能指标（IPC、Miss Rate、GFLOPS、延迟、能耗）

**Q4.** 
- 使用多次实验取平均值（≥ 3 次）
- 报告标准差或 95% 置信区间
- 对于仿真结果，确保 warm-up 充分（跳过前 10-20% 指令）

**Q5 (gem5 CPU).** 参数扫描范围需有实际意义：发射宽度 1-8（8+ 很少有意义），
ROB 32-256（再大设计复杂度极高），物理寄存器 64-512。

**Q6 (gem5 CPU).** 使用检查点（checkpoint）跳过启动阶段；限制模拟指令数
（如 10M 指令采样准确度可接受）；并行运行多个 gem5 实例。

**Q7.** 常见风险与应对：
| 风险 | 缓解方案 |
|------|---------|
| gem5 编译失败 | 使用 Docker 预配置镜像 |
| CUDA 环境不可用 | 使用 Google Colab 免费 GPU |
| 项目范围过大 | 缩小研究问题数量，聚焦核心对比 |
| 仿真时间远超预期 | 减少参数点、使用采样技术 |

**Q8.** 优先级：核心性能数据 > 可视化图表 > 详细文字分析 > 额外对比实验。
确保至少一个完整的设计空间对比结果。
