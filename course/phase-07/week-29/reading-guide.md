# Week 29 阅读指南：仓库级计算机（Warehouse-Scale Computers）

> 教材章节：Ch6 §6.1-6.5 | 核心主题：大规模数据中心架构与成本模型

## 学习目标

- 理解 WSC（Warehouse-Scale Computer）与传统服务器的本质区别
- 掌握总拥有成本（TCO）模型的构成与计算方法
- 理解 PUE（Power Usage Effectiveness）及其优化策略
- 掌握能量比例性（Energy Proportionality）的概念
- 理解大规模系统中的故障处理策略与尾延迟（Tail Latency）

## 核心概念

### 1. WSC 架构特点
- **规模**：数万至数十万台服务器，统一管理和调度
- **同质性**：硬件/软件栈统一，降低运维复杂度
- **网络为核心**：Clos 网络拓扑，对分带宽至关重要
- **存储层次**：本地 SSD → 分布式存储 → 冷存储

### 2. TCO 总拥有成本模型
- **CapEx（资本支出）**：服务器、网络设备、数据中心建设
- **OpEx（运营支出）**：电力、冷却、运维人员、网络带宽
- **关键公式**：TCO = CapEx + Σ(OpEx_t / (1+r)^t)，考虑折现率 r

### 3. PUE 与能效
- **PUE = 总能耗 / IT 设备能耗**
- 理想 PUE ≈ 1.0，业界领先约 1.1-1.2
- **PUE 分解**：冷却、配电损耗、照明等非 IT 负载

### 4. 能量比例性
- **定义**：服务器在负载为 x% 时应消耗 x% 的峰值功率
- **现实**：多数服务器空闲时仍消耗 30-50% 峰值功率
- **影响**：造成大量能源浪费，推动低功耗状态设计

### 5. 故障处理与尾延迟
- **故障是常态**：MTTF × 服务器数量 → 每天都有故障
- **故障层次**：磁盘故障 > 内存错误 > 网络故障 > 服务器宕机
- **尾延迟放大器**：并行请求中，最慢的响应决定整体延迟
- **对冲请求（Hedged Requests）**：向多个副本发请求，取最先返回的

### 6. MapReduce 编程模型（Ch6 §6.2）

WSC 的典型计算框架是 MapReduce，它将大规模数据处理抽象为三个阶段：

- **Map**：输入数据被划分为独立分片（splits），每个 Map 任务处理一个分片，输出中间键值对
- **Shuffle / Sort**：系统自动按 key 对中间结果排序、分组，将相同 key 的数据传输到同一 Reduce 节点——这是网络 I/O 最密集的阶段
- **Reduce**：每个 Reduce 任务对一组 key 的 values 执行归约操作，输出最终结果

数据局部性（Data Locality）是关键优化：调度器优先将 Map 任务分配到数据所在的节点（HDFS 本地读取），避免跨网络传输 TB 级输入数据。MapReduce 的设计哲学——将计算移动到数据所在位置而非移动数据——直接影响了 WSC 的网络拓扑和存储层次设计。

## 关键公式速查

| 公式 | 含义 |
|------|------|
| TCO = CapEx + NPV(OpEx) | 总拥有成本 |
| PUE = P_total / P_IT | 电能使用效率 |
| Availability = MTTF / (MTTF + MTTR) | 可用性 |
| Tail Latency = P99 / P50 | 尾延迟放大比 |
